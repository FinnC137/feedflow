## 2026-05-02：项目启动 & 架构定型

### 背景

LT_News 运行良好，但它是单信源（YouTube LT视界）的单点方案。
用户提出两个方向的需求：
1. 输出端从 Flomo 迁移到 RSS
2. 输入端从 YouTube 扩展到多信源（B站、博客等）

### 决策 1：新项目，不修补 LT_News

| 选项 | 评估 |
|------|------|
| 在 LT_News 上修补 | ❌ 项目定位已变（单点→通用），修补会导致代码腐化 |
| **开新项目 FeedFlow** | ✅ 独立开发，并行运行，稳定后 LT_News 退役 |

**理由**：这个项目本质上已经进化到不是 LT News 的每日采集器，是一个更通用的高级能力。

### 决策 2：按信源家族分采集器

| 选项 | 评估 |
|------|------|
| 一个万能采集器 | ❌ YouTube 和 B站采集流程完全不同，硬捏成一个函数是错误抽象 |
| **按家族分（youtube.py / bilibili.py / rss.py）** | ✅ 相同流程的信源归一类，新增信源只需写一行配置 |

**教训来源**：ProSummary 项目 — B站字幕 API 强制登录是平台策略（不可移除），不是工具问题。
不同平台不能硬捏成一个万能函数。

### 决策 3：统一调度，不搞细粒度

| 选项 | 评估 |
|------|------|
| 每个信源独立 cronjob | ❌ cron 列表膨胀，维护成本高 |
| 高频检测（30分钟） | ❌ 时效性不强，搞复杂了 |
| **统一 cronjob 每天 7:10** | ✅ 佛系、不搞复杂、设完隐身 |

### 决策 4：三档输出级别（per-source configurable）

| 级别 | 产出 | 适用 |
|------|------|------|
| heavy | 清洗后完整逐字稿 | 需要研究原文 |
| default | 结构化摘要（同当前 LT_News） | 日常看新闻 |
| light | 精简摘要（3-5句话） | 快餐式浏览 |

### 决策 5：Schema 驱动的网页配置编辑器

| 选项 | 评估 |
|------|------|
| 先内嵌在 FeedFlow 中 | ✅ 先实打实干完活，等第二家项目出现再提取为独立框架 |
| 一开始就做独立框架 | ❌ 过度工程 — 不知道第二家具体需求前，抽象接口一定是错的 |

### 设计原则（继承 ProSummary）

| 原则 | 来源 | 在 FeedFlow 中体现 |
|------|------|-------------------|
| 轻依赖优先 | ProSummary | YouTube 采集主方案用 baoyu-youtube-transcript（零 API Key） |
| 降级链兜底 | ProSummary | 每个 fetcher 预留 3 层：主方案 → 备选 → 兜底 |
| 渐进固化 | ProSummary | 先用 CLAUDE.md 指令驱动，稳定后固化为 Python CLI |
| 凭证统一 | ProSummary | 所有敏感信息集中在 secrets.json |

### 模型选型讨论

当前栈：DS V4 Flash（日常）+ DS V4 Pro（复杂实现）+ Claude Sonnet 4（架构/指令遵循）

**结论**：开发阶段 Flash 够用，Pro 在实现核心逻辑时更有价值。
考虑通过 OpenRouter 引入 Opus 4 系列作为上限救场（待评估）。
