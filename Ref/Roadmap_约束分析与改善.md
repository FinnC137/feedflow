## 这份文档在说什么

ProSummary 是一个私人信息处理工具。它目前能用、体验也不错，但运行起来依赖一条很长的链：Claude 订阅 → Claude Code 保持运行 → 电脑不关机 → B站账号 → Flomo PRO → 一堆 Skill → 网络...

这份文档做三件事：

1. **盘清所有约束**——把"它需要什么才能跑"列一个完整清单，不遗漏

2. **按移除难度分级**——哪些一小时搞定，哪些需要架构改动

3. **给出改善方案**——前提是**不降低目前已有的体验品质**

### 核心目标

所有改善都服务于同一个方向：

> **让 ProSummary 从"只能在特定条件下跑的个人原型"逐步演变为"一个可以被任何 Agent 平台调用的功能模块"。**

具体来说：

* **降低运行门槛**：减少"它必须在才能跑"的东西

* **提升可移植性**：不绑定 Claude Code，OpenCode / Hermes Agent / Cursor / cron 都能调用

* **保持优雅**：这是私人工具，不追求产品化，但要干净、鲁棒、可维护

### 架构认知（讨论中确立）

ProSummary 的本质定位是**功能模块**，不是应用。它对外暴露一个接口：

```
prosummary(url_or_text) → 抓取 + 总结 + 推送
```

两个关键分层：

* **"谁调度我"和"我内部消耗谁的算力"是两件独立的事**

* 外层 Agent（Claude Code / OpenCode / Hermes）负责调度，用各自平台的 LLM

* ProSummary 内部如果需要 LLM（总结、异常决策），走自己 `secrets.json` 里配的 API，与外层无关

```
外层（调度者）              内层（ProSummary 功能模块）
┌──────────────────┐     ┌──────────────────────────┐
│  Claude Code     │     │  抓取（脚本，零LLM）        │
│  OpenCode        │     │  降级决策（脚本+LLM补位）    │
│  Hermes Agent    │ ──→ │  内容总结（LLM）            │
│  Cursor          │     │  归档 + 推送（脚本，零LLM）  │
│  cron 定时器     │     │                            │
│                  │     │  内部 LLM 通过统一接口调用：  │
│  各自的 LLM 算力  │     │  provider=deepseek/claude  │
└──────────────────┘     └──────────────────────────┘
     调度算力 ≠ 执行算力
```

LLM 在 ProSummary 内部不只是做总结——在脚本遇到预设路径无法处理的异常时，LLM 作为"安全员"介入做运行时决策（短链解析、降级判断、编码错误诊断等）。脚本处理 happy path，LLM 处理 edge case。

## 按移除难度分级 + 改善方案

### 🟢 第一档：轻松可移除（1-2小时内，零成本）

#### 1.1 凭证统一存放 ✅ 已完成

|        |                                                                           |
| ------ | ------------------------------------------------------------------------- |
| **原状** | Flomo Webhook 在 `.env`，B站 Cookie 在 `~/.bilibili-cli/credential.json`，分散两处 |
| **改善** | 统一到 `Finn_ProSummary/secrets.json`，一个文件管理所有敏感信息                           |
| **效果** | 迁移环境时只需复制一个文件；未来 LLM API Key 也预留了字段                                       |
| **状态** | ✅ 2026-04-27 已完成。`.gitignore` 已配置保护                                       |

#### 1.2 暗号触发 → 正式 Skill

|        |                                                                               |
| ------ | ----------------------------------------------------------------------------- |
| **原状** | 用户必须说"Pro总结"暗号，Claude 在 CLAUDE.md 里匹配关键词                                      |
| **改善** | 注册为正式 Skill，通过 URL pattern 自动识别触发（bilibili.com / youtu.be / xiaoyuzhoufm.com） |
| **效果** | 贴个链接就自动跑，不用记暗号；其他 Agent 平台也能直接调用                                              |
| **状态** | ✅ 2026-04-27 已完成                                                              |

#### 1.3 PYTHONIOENCODING 环境变量固化 ✅ 已完成

|        |                                                                   |
| ------ | ----------------------------------------------------------------- |
| **原状** | bilibili-cli 每次运行都要手动加 `PYTHONIOENCODING=utf-8`，否则 Windows GBK 崩溃 |
| **改善** | 写入 `~/.bashrc`：`export PYTHONIOENCODING=utf-8`                    |
| **效果** | 永久解决，不再踩坑                                                         |
| **状态** | ✅ 2026-04-27 已完成。新 session 自动加载，所有 Python CLI 工具受益                |

### 🟡 第二档：中等难度（半天到一天工作量）

#### 2.1 多输出端（降低 Flomo PRO 的必要性）

|        |                                                                                                                   |
| ------ | ----------------------------------------------------------------------------------------------------------------- |
| **原状** | 输出只有 Flomo 一条路，Flomo PRO 到期 = 推送能力丧失                                                                              |
| **改善** | 新增输出选项（可并存）：**A. 飞书文档**（已有 lark-cli，半小时接入）**B. 本地 Obsidian / Markdown**（零成本，Files/ 归档已在做）**C. Notion API**（免费额度大） |
| **效果** | Flomo 依然是默认主力输出（不动），但不再是唯一出口。任一通道可用即可完成任务                                                                         |
| **前提** | 不降低体验——Flomo 用着好就继续用，新增通道是"多一条路"而非"替换"                                                                            |
| **状态** | ⏳ 待实施。优先级：飞书（已有基础设施）> Notion > 其他                                                                                 |

#### 2.2 B站 Cookie 健康检查

|        |                                                     |
| ------ | --------------------------------------------------- |
| **原状** | bilibili-cli 有 7 天 TTL auto-refresh，但偶尔失效需手动重新提取    |
| **改善** | 写一个简单的健康检查：运行 `bili whoami` 检测凭证有效性，失败时主动提醒（飞书消息通知） |
| **效果** | 减少"跑到一半发现 Cookie 过期"的情况                             |
| **状态** | ⏳ 待实施                                               |

#### 2.3 YouTube 降级链补强

|        |                                                                     |
| ------ | ------------------------------------------------------------------- |
| **原状** | baoyu-youtube-transcript 失败 → NotebookLM（重依赖）→ 粘贴长文                 |
| **改善** | 在 baoyu 和 NotebookLM 之间插一层：`yt-dlp --write-subs`（已安装，零额外依赖）直接下载字幕文件 |
| **效果** | 降级链更轻，减少对 NotebookLM 的触发频率                                          |
| **状态** | ✅ 2026-04-27 已完成                                                    |

### 🟠 第三档：需要架构改动（1-3天工作量）

#### 3.1 编排层脚本化（核心改动）

|              |                                                                                                                                                    |
| ------------ | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| **原状**       | Claude 既是编排者又是执行者。每次都要读 CLAUDE.md ~160 行指令（~3000 tokens），理解后临时组装命令链                                                                                |
| **改善**       | 将"抓取→降级→归档→推送"固化为 `prosummary.py`：- Happy path 由脚本自动走完（URL解析、调Skill、归档、推Flomo）- Edge case 抛给 LLM API 做运行时决策（降级判断、异常诊断）- 总结调用 LLM API（provider 可配置） |
| **效果**       | ① Claude 不在也能跑 80% 的流程② 可被其他 Agent 平台作为函数调用③ 可接 cron 定时触发（人不在场）④ 每次省 ~3000 tokens（不用读 CLAUDE.md）                                                   |
| **LLM 角色变化** | 从"全程驾驶员"变为"安全员"——脚本开自动驾驶，LLM 只在施工路段接管方向盘                                                                                                           |
| **触发时机**     | 连续 N 次执行都不需要改流程时，说明可以固化了                                                                                                                           |
| **状态**       | ⏳ 待实施。当前仍在原型验证期，有意不提前固化                                                                                                                            |

#### 3.2 LLM 统一接口（全流程可切换）

|          |                                                                                                                |
| -------- | -------------------------------------------------------------------------------------------------------------- |
| **原状**   | 整条管线绑死 Claude——Claude 既提供编排智能，又提供总结能力                                                                          |
| **改善**   | `secrets.json` 中已预留 `llm` 段，脚本化时实现统一调用接口：`llm.chat(prompt, text)` 内部根据 `provider` 路由到 Claude / DeepSeek / Kimi |
| **关键约束** | **全局只有一个 LLM 出口**，不做"A模型负责总结、B模型负责决策"的多模型并行。换就是整体换，一行 config 切换，避免多模型维护成本                                      |
| **代码量**  | 核心代码约 10 行（OpenAI-compatible API 格式，DeepSeek/Kimi/Gemini 都兼容）                                                  |
| **验证成本** | 半小时内可跑通 DeepSeek API 验证                                                                                        |
| **状态**   | ⏳ 待实施。`secrets.json` 中的 `llm` 字段已预留                                                                            |

#### 3.3 Skill 统一打包

|        |                                             |
| ------ | ------------------------------------------- |
| **原状** | 4 个 Skill 分散安装（npm/pip/手动），版本管理靠人记          |
| **改善** | 统一为一个 `prosummary` Python 包，内含所有 scraper 模块 |
| **效果** | `pip install prosummary` 一条命令搞定全部依赖         |
| **状态** | ⏳ 低优先级。等脚本化完成后再考虑                           |

### 🔴 第四档：不可移除（接受并优化体验）

#### 4.1 B站强制登录

|            |                                        |
| ---------- | -------------------------------------- |
| **本质**     | B站字幕 API 强制要求登录态，这是平台策略，不是工具的问题        |

| **无法再优化**  | 除非 B站开放匿名字幕 API（不太可能）                  |

#### 4.2 网络依赖

|           |                              |
| --------- | ---------------------------- |
| **本质**    | 所有信息源都是在线内容，必须联网             |
| **已做到的**  | 抓取后本地归档（Files/ 目录），已处理内容离线可查 |
| **无法再优化** | 新内容必须联网，这是本质约束               |

#### 4.3 Whisper 本地资源消耗

|          |                                                    |
| -------- | -------------------------------------------------- |
| **本质**   | 小宇宙播客转录用 faster-whisper，长音频吃 CPU/GPU               |
| **可选优化** | 换云端 ASR（阿里云/讯飞/OpenAI Whisper API），但引入新 API Key 依赖 |
| **当前判断** | 与"轻依赖优先"原则矛盾，暂不动。tiny 模型够用，base 模型应急               |

## 约束本质分类（速查）

| 类型       | 约束项                              | 可解性                |
| -------- | -------------------------------- | ------------------ |
| **平台锁定** | Claude 订阅、Claude Code 运行时、电脑不关机  | 脚本化 + LLM 可切换 → 解除 |
| **付费墙**  | Flomo PRO、Claude 订阅              | 多输出端 + 换 LLM → 绕过  |
| **信任成本** | B站Cookie、Flomo Token、LLM API Key | 统一加密存储 → 已优化 ✅     |
| **不可移除** | 网络、B站强制登录、平台反爬                   | 接受，优化体验            |
